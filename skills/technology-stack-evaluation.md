# Technology Stack Evaluation

## Purpose
Audit the current martech stack for fit, integration, and utilisation, identify gaps and redundancies, and define requirements for new technology investments.

---

## When to Use This Skill
- Annual martech review
- Before a new technology purchase (to prevent redundancy)
- After a merger, acquisition, or restructure that merged stacks
- When the stack feels complex, expensive, or underutilised
- Preparing a technology investment case for finance

---

## Briefing Checklist
Before starting, confirm:
1. **Business objective** — Cost reduction, capability improvement, or rationalisation?
2. **Scope** — Full martech stack or a specific category (CRM, MAP, analytics)?
3. **Stakeholders** — Who owns each tool, and who needs to be involved in decisions?
4. **Budget context** — Is there a target spend reduction or a new capability budget?
5. **Integration landscape** — What are the core systems of record the stack must connect to?

## Process

### Step 1 — Inventory the Current Stack
Document every tool in the marketing technology ecosystem:

| Tool | Category | Owner | Annual Cost | Users | Core Function | Integration Points |
|---|---|---|---|---|---|---|
| — | — | — | £— | — | — | — |

Categories to cover:
- CRM (Customer Relationship Management)
- MAP (Marketing Automation Platform)
- Web analytics and tag management
- Paid media management and bidding
- SEO and content tools
- Social media management
- ABM and intent data
- Chat, conversational marketing
- Email marketing (if separate from MAP)
- Events and webinar platforms
- Video hosting and production
- Project management and collaboration
- Reporting and BI

### Step 2 — Assess Each Tool Against Four Dimensions

| Dimension | Questions to Ask |
|---|---|
| **Fit** | Does this tool solve the problem it was bought for? Is it the right solution for our current scale? |
| **Utilisation** | What % of features are actively used? How many users log in monthly? |
| **Integration** | Does it connect cleanly to other tools in the stack? Are there manual workarounds because of integration gaps? |
| **Cost vs. value** | Is the cost justified by the measurable value delivered? Is there a cheaper alternative with sufficient capability? |

Rate each tool: High / Medium / Low across all four dimensions.

### Step 3 — Identify Redundancies
Look for:
- Tools that do the same thing (e.g. two email platforms, two analytics tools)
- Features in existing tools that overlap with standalone tools (e.g. MAP with landing page builder vs. separate landing page tool)
- Zombie tools — contracted but unused, or used by fewer than 3 people

Calculate total annual cost of redundant tools — this is the savings case.

### Step 4 — Identify Gaps
Map business requirements that are not met by the current stack:
- Where are people using spreadsheets or manual processes because no tool exists?
- What capabilities does the roadmap require that aren't in the current stack?
- Where is integration missing, causing data silos or manual data entry?

Build a gap register:

| Capability Gap | Business Impact | Urgency | Estimated Cost to Solve |
|---|---|---|---|
| — | — | High/Med/Low | £— |

### Step 5 — Define the Target Stack Architecture
Design the desired state:
- Core tools to keep and fully utilise
- Tools to consolidate or replace
- New capabilities to add
- Integration architecture: how data flows between systems

Map the data flow: Lead capture → Enrichment → CRM → MAP → Analytics → Reporting

Identify the system of record for each data type (contact, account, deal, attribution).

### Step 6 — Build the Business Case
For rationalisation: annual cost saved, migration cost, net saving over 3 years
For new investment: capability enabled, revenue impact, implementation cost, payback period
For consolidation: consolidation savings vs. migration cost and risk

## Output Formats

### Tech Stack Audit Report
- Full tool inventory with costs and utilisation
- Assessment matrix (fit, utilisation, integration, value)
- Redundancy analysis with savings opportunity
- Gap register
- Recommended stack changes with rationale

### Requirements Document (for new tools)
- Problem statement
- Must-have capabilities
- Integration requirements
- Data and security requirements
- Budget range

### Vendor Comparison Matrix
- Shortlisted tools scored against requirements
- Integration capability assessment
- TCO comparison
- Recommended selection with rationale

---

## Common Mistakes to Avoid
- **Buying tools to solve process problems.** New technology doesn't fix broken processes — it automates them.
- **Evaluating tools in isolation.** A tool that doesn't integrate with the stack creates more problems than it solves.
- **Ignoring utilisation data.** A well-priced tool nobody uses is still wasted spend.
- **Chasing features over fundamentals.** A clean CRM and a working MAP outperform a sophisticated stack that nobody trusts.
- **No ownership model.** Every tool needs a named internal owner responsible for its performance and renewal decision.

---

## Quality Check
Before submitting, ask:
- [ ] Is every tool in the stack inventoried with cost and ownership?
- [ ] Are redundancies identified with a quantified savings opportunity?
- [ ] Is a gap register built from business requirements, not feature wishlists?
- [ ] Is the target architecture defined with data flows mapped?
- [ ] Does every tool have a named internal owner?
