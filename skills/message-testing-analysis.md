# Message Testing Analysis

## Purpose
Design and analyse message tests across headlines, value propositions, CTAs, and creative to generate messaging recommendations grounded in evidence rather than opinion.

---

## When to Use This Skill
- Developing or refreshing core messaging and positioning
- Optimising landing pages, ads, or email subject lines
- Validating a new value proposition before a campaign launches
- Resolving internal disagreements about what message will perform best

---

## Briefing Checklist
Before starting, confirm:
1. **Business objective** — Conversion improvement, positioning validation, or message development?
2. **What's being tested** — Headline, value proposition, CTA, body copy, or creative concept?
3. **Channel** — Email, paid ad, landing page, website hero? Channel determines test design.
4. **Success metric** — Click rate, conversion rate, engagement, or response rate?
5. **Volume** — What is the available sample size? Determines statistical feasibility.

---

## Process

### Step 1 — Define the Test Hypothesis
Every test must start with a hypothesis, not a question:

**Structure:** "We believe [changing X to Y] will [increase metric Z] because [customer insight or assumption]."

**Example:** "We believe changing the hero headline from feature-led ('Automate your reports') to outcome-led ('Get your Monday back') will increase landing page conversion rate because our ICP research shows time-saving is the primary purchase driver."

Document the hypothesis before building the test — it prevents post-hoc rationalisation.

### Step 2 — Design the Test

| Element | Decision |
|---|---|
| Control | What is the current baseline? |
| Variant(s) | What is being changed — and only that |
| Sample split | 50/50 for two variants; adjust for more |
| Minimum sample size | Calculate based on baseline conversion rate and MDE |
| Test duration | Long enough for statistical significance; at least 1–2 full business cycles |
| Success metric | Single primary metric — avoid multiple primary metrics |

**Minimum Detectable Effect (MDE):** Define the smallest improvement that would be meaningful before testing. A 1% improvement on a 30% baseline is different from a 1% improvement on a 2% baseline.

### Step 3 — Run the Test with Discipline
- Change one variable at a time in A/B tests
- Run tests for the full planned duration — do not stop early when results look good
- Ensure audience is randomly split and not self-selecting
- Track confounding variables (seasonality, other campaigns, algorithm changes)

### Step 4 — Analyse Results

| Metric | Control | Variant | Absolute Difference | Relative Lift | Confidence Level |
|---|---|---|---|---|---|
| Primary KPI | —% | —% | — pp | —% | —% |

Statistical significance threshold: 95% confidence minimum for decisions; 90% acceptable for directional learnings.

Assess:
- Did the variant beat the control?
- Is the result statistically significant, or could it be noise?
- Is the effect size large enough to be practically meaningful?
- Are there any segment-level differences that cut across the overall result?

### Step 5 — Synthesise Into Messaging Recommendations
Test results inform but don't replace judgment:

| Test Result | Messaging Implication |
|---|---|
| Outcome-led headline outperformed feature-led by 28% | Lead with customer outcome, not product capability, across all acquisition copy |
| CTA "Start free trial" outperformed "Get started" by 12% | Specificity and risk-reduction language outperforms generic CTAs — apply across web |
| No significant difference between variants | The message axis tested doesn't move this metric — test a different variable |

### Step 6 — Document and Systematise
Log every test in a message testing repository:
- Hypothesis
- Test design
- Results with confidence level
- Recommendation and where it was applied
- Next test to run

---

## Output Formats

### Message Test Plan
- Hypothesis and rationale
- Control and variant description
- Test design (split, duration, sample size, metric)
- Expected timeline and readout date

### Results Analysis
- Quantitative results table
- Statistical significance assessment
- Segment-level cuts (if relevant)
- Qualitative interpretation

### Messaging Recommendation Brief
- What the test found
- What it means for messaging strategy
- Where the winning message should be applied
- Next test to run

---

## Common Mistakes to Avoid
- **Testing without a hypothesis.** A test without a hypothesis is a guessing game, not a learning.
- **Stopping tests early.** Peeking at results and stopping when significant introduces false positive risk.
- **Testing too many variables at once.** Multi-variable tests (without full factorial design) make results uninterpretable.
- **Ignoring practical significance.** A result can be statistically significant and commercially irrelevant.
- **Not documenting learnings.** Tests that aren't logged don't compound — the organisation re-learns the same things.

---

## Quality Check
Before submitting, ask:
- [ ] Is there a documented hypothesis with a clear rationale?
- [ ] Is the test designed to isolate a single variable?
- [ ] Is the sample size sufficient for statistical significance at the planned MDE?
- [ ] Are results interpreted with both statistical and practical significance in mind?
- [ ] Are findings logged in the shared testing repository?
